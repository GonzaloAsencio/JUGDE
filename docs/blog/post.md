# Building a Rules Judge for a Trading Card Game with RAG: What I Learned

---

## The Hook

I gave the system a question that should have been easy: "What does Quick Strike do?"

The rulebook had a clear, verbatim answer. The paragraph was three sentences. The word "Quick Strike" appeared twice in the first sentence. And the system returned the wrong answer — a paragraph about combat phase timing that had nothing to do with the keyword.

The model wasn't lying. The retriever was.

That moment kicked off three weeks of iterating on retrieval, building an honest evaluation setup, and making a handful of architectural decisions I'd make differently if I started over. This is the honest writeup.

---

## The Problem

Large language models are surprisingly good at answering general knowledge questions about trading card games. You can ask ChatGPT "how does summoning sickness work in Magic: The Gathering" and get a reasonable answer. But there are two categories where general LLMs fall apart for TCG rules work:

First, unofficial or niche games. Riftbound is not a household name. The LLM has essentially no training data about it. Ask it a Riftbound-specific rule and it will confidently hallucinate something plausible-sounding.

Second, errata and version-specific rules. TCGs update their rules constantly. A plain LLM has no concept of which rulebook version is canonical for a given tournament format.

The obvious solution is Retrieval-Augmented Generation: embed the official rulebook, retrieve the relevant chunks at query time, and force the model to answer only from what it retrieves. If the context does not contain the answer, the model defers rather than fabricates.

That framing sounds simple. The interesting part is building it so it actually works — which means defining "actually works" before writing a line of code.

---

## The Approach

I started from the evaluation spec, not the implementation. Before writing any retrieval code, I wrote down four questions I wanted to be able to answer with data:

1. Does the baseline (vector-only) retrieval surface the right chunks for the eval set?
2. Does hybrid retrieval improve on baseline? By how much?
3. Do any query categories systematically fail?
4. Does entity resolution (matching card names to card text) help card-specific queries?

This forced me into an eval-first discipline. Every architectural decision had to be tied to something measurable. If I couldn't define a metric for it, I deferred it.

The downside is that I'm shipping a README with TBD in the results table. The eval set exists; the RAGAS runs haven't happened yet. I'd rather have honest TBDs than numbers I invented.

## Building the Eval Set

The eval set is 40–60 hand-curated question-answer pairs, not generated by an LLM.

I considered the LLM-generated approach: feed the rulebook to GPT-4, ask for 200 questions, done. I rejected it for a specific reason: LLM-generated questions tend to ask about things that are easy to retrieve — they mirror the document structure. A real user asking about a corner case in simultaneous combat resolution is not going to phrase the question the way the rulebook phrases the rule. Human-written questions capture the vocabulary mismatch between user intent and rulebook text. That mismatch is exactly what the retriever needs to handle.

The eval set is broken into five categories: factual (single-citation answers), multi-step (requires synthesizing two or more rules), card-specific (keyword or mechanic lookup by name), edge case (ambiguous or obscure rules), and adversarial (prompt injection attempts and off-topic queries). Each question has a reference answer and the expected section(s) that should appear in the citations.

Forty curated questions with known-good answers are more useful than two hundred mediocre ones. Quality beats quantity when your eval budget is finite and you're the only person running evals.

## Baseline: Simple RAG

The first implementation was deliberately minimal: encode the query with bge-m3, retrieve the top-5 most similar chunks by cosine distance from pgvector, pass them to Gemini 2.0 Flash with a grounding prompt, return the answer with citations.

The grounding prompt told the model to: answer only from the provided chunks, cite the section for every claim, and say "I cannot answer this" rather than hallucinate when the context didn't contain the answer.

Baseline results: TBD — to be filled once the full eval run completes. The qualitative observation was that semantic paraphrase queries worked reasonably well. "How does the damage step work?" returned relevant chunks. "What are the restrictions on playing units during your opponent's turn?" worked. But exact-token queries — card names, rule number references, errata identifiers — were unreliable. The "Quick Strike" failure I opened with was the clearest example.

---

## Hybrid Retrieval: Fixing the Exact-Token Problem

The root cause of the exact-token failures was straightforward: dense embeddings generalize. "Quick Strike" and "fast attack" might have high cosine similarity because the embedding model has seen enough text to know they're semantically related. But if the rulebook uses the term "Quick Strike" and the query uses "Quick Strike", you want an exact match — not a semantic approximation.

The fix was to add a second retrieval signal: Postgres full-text search using `to_tsvector('simple', content) @@ plainto_tsquery('simple', query)`. FTS does not care about semantics — it does exact token matching with basic stemming. "Quick Strike" in the query matches "Quick Strike" in the corpus directly.

I fused the two result lists using Reciprocal Rank Fusion. RRF works like this: for each chunk, compute a score equal to the sum of `1 / (rrf_k + rank)` across every result list where the chunk appears. Chunks that appear in both the dense and FTS lists get a bonus. `rrf_k = 60` is the standard constant from the original RRF paper — it smooths the contribution of results ranked low in one list.

The tie-break rule is: when two chunks have identical RRF scores, the chunk from the dense vector list wins. This is not because dense is "better" — it's because semantic relevance is the primary signal, and exact-token matching is a supplement.

The configuration in production: `top_k_fetch = 15` from each signal, fused to `top_k = 5`. That means 30 rows fetched from Postgres before trimming to 5. The latency cost is approximately 20% higher on the retrieval step. Worth it.

## The Entity Resolution Decision

There is a class of queries where the user is asking about a specific card: "What does Veilbreaker do?" The answer requires card text, not just rulebook text. The rulebook might reference the card by name but not reproduce its full ability text.

The hypothesis was: detect card names in the query, look up the card text, inject it into the prompt context alongside the rulebook chunks.

I built a forward hook — `card_mentions: list[str] | None` flows through `pipeline.answer_question()` and is used in the cache key. But I did not build the UI autocomplete, the cards database, or the prompt injection.

The reason was data-driven. The threshold in the evaluation spec was: build entity resolution if card-specific query failures exceed 20% of the card-specific category in the eval set. Without running the eval, I had no number. I was being asked to add significant complexity (a separate ingestion pipeline, a cards database, frontend @mention UX) based on a hypothesis.

I chose to defer and let the data decide. If the eval shows card-specific failure rate above 20%, I build it. If it's below 20%, the rulebook chunks are sufficient and I don't need the complexity. The threshold is explicit and the decision is reversible.

The lesson here is that "it seems like it would help" is not an architectural justification. Data is.

## What Surprised Me

**The structlog / PrintLoggerFactory incompatibility.** The project uses `structlog` for structured JSON logging. On first startup in a fresh environment, the logger initialization threw a confusing error because `PrintLoggerFactory` was being called with keyword arguments that changed between structlog versions. The fix was trivial but the debugging was frustrating because the error message pointed nowhere near the actual cause. Pinning the structlog version in `requirements.txt` resolved it, but it cost an afternoon.

**Gemini 2.5 Flash thinking mode latency.** I briefly tested Gemini 2.5 Flash with thinking mode enabled, expecting it to produce better grounded answers for multi-step rule questions. The quality improvement was marginal. The latency was not: thinking mode adds roughly 10–15 seconds per request. For an interactive Q&A interface where users expect an answer in under 3 seconds, that is a non-starter. I reverted to 2.0 Flash immediately. The lesson: for retrieval-augmented tasks where the context is already provided, thinking mode does not buy much — the LLM is not reasoning from first principles, it's reading and summarizing.

**Cache hit rate was higher than expected.** Many rule questions are semantically similar even when phrased differently. "Can I block on my opponent's turn?" and "Is blocking allowed on the opponent's turn?" hit the same cache entry because the exact-match cache key normalizes question whitespace and lowercases before hashing. I had expected low hit rates for a rules judge with diverse questions, but in practice the hit rate during manual testing was substantial. The implication for the roadmap is that semantic caching (ANN over question embeddings to catch paraphrase duplicates) could push hit rates even higher and meaningfully reduce Gemini token spend.

## What I'd Do Differently

**Define the eval schema before writing ingestion code.** I built the ingestion pipeline before I had a final eval format. When I later designed the eval set and RAGAS integration, I had to retrofit metadata onto chunks that were already stored. Having the evaluation schema first would have shaped what metadata I stored at ingestion time — particularly chunk-level source type tags that I ended up adding in a later migration.

**Commit eval run JSONs earlier.** The `backend/data/eval_runs/` directory contains only a `.gitkeep`. The eval set is designed, the pipeline is running, but I haven't committed a single RAGAS output. This means the README has a TBD results table at launch. I should have run at least a baseline eval on a small set early, even if the numbers were rough. Rough numbers are better than no numbers when you're trying to make architectural decisions.

**Tighter logging from day one.** Adding Langfuse and Sentry observability late in the process meant I had to retrofit trace IDs and structured fields into existing log statements. If I had designed the logging schema before writing the pipeline, every log event would have had consistent fields from the start.

## Update — 2026-06-22

This post shipped with TBD results and named RAGAS as the eval framework. Two things changed after publishing, and I'd rather correct the record than leave it stale:

- **The eval did run.** A baseline against the production hybrid pipeline: ~25% correct (35% correct+partial), 14% retrieval recall on the 20-question set. Low — and that's the honest starting point I now optimize from. Rough numbers beat no numbers.
- **It does not use RAGAS.** Evaluation was the last thing built, so I shipped a dependency-light LLM-as-judge harness (verdict correct/partial/wrong) plus a deterministic retrieval-recall check, instead of integrating RAGAS. The full reasoning — and what that costs me (no measured faithfulness) — is in [ADR-006](../adrs/ADR-006-eval-framework.md). RAGAS is now optional future work.

The retrospective above stands as written; this is just the follow-through.
